First: How Real Dental Systems Actually Work (Mentally)

Forget schema for a moment.

Real systems revolve around ONE flow:
 
Chart → Diagnose(Current Condition, Exiting Condition ) → Treatment Plan → Perform → Update Chart
_________________________________________________________________
CORE ARCHITECTURE (Dentrix-Level Mental Model)

Here’s the architecture you should aim for:

Dental Chart (STATE)
    ↓
Findings / Conditions (DIAGNOSIS)
    ↓
Treatment Plan (INTENT)
    ↓
Planned Procedures (WHAT to do)
    ↓
Visits (WHEN to do)
    ↓
Procedure Sessions (ACTUAL execution)
    ↓
Chart Updates (RESULT)

existing work

2. The 5 CORE ENTITIES YOU MUST GET RIGHT

If these are correct → your system becomes world-class.

1. 🦷 Dental Chart (STATE ENGINE)

Instead of:

ToothRecord
ToothProcedure
scattered updates

👉 You need a layered chart model

Recommended model:
ChartEntry {
  id
  patientId

  toothNumber Int?
  surfaces ToothSurface[]

  type ENUM:
    CONDITION   // decay, fracture
    EXISTING    // filling, crown already there
    PLANNED     // from treatment plan
    COMPLETED   // executed

  procedureId?
  treatmentProcedureId?
  procedureSessionId?

  status ENUM:
    ACTIVE
    SUPERSEDED

  createdAt
}
Why this matters:
Chart becomes history + current state
You can:
render UI easily
track progression
undo safely
____________________________________________
2. 🧾 Treatment Plan = “Clinical Intent”

This is where Dentrix Ascend is strong.

Key concept:

👉 Plans are grouped into cases/options

Dentrix supports:

multiple plans (ideal vs alternative)
You should support:
TreatmentPlan
  ├── version
  ├── caseType (PRIMARY | ALTERNATIVE)
  └── TreatmentProcedure[]
3. 🧩 TreatmentProcedure = “Atomic Clinical Unit”

This is NOT execution.

This is:

“We plan to do X on tooth Y”

___________________________________________
Must include:
target (tooth/surface)
clinical notes
sequencing
grouping (phase)
Missing in your system:

👉 Procedures grouped by visit/phase

Dentrix literally supports:

grouping procedures by visit sequence templates
You should add:
phase String // "Phase 1", "Surgery"
plannedVisitIndex Int // visit 1, 2, 3
___________________________________________
4. 📅 Visit = “Time Container”

This is correct in your system already.

But in real systems:

👉 Visits are NOT random
They are derived from the plan.

Key behavior:
One visit = multiple procedures
One procedure = can span visits
